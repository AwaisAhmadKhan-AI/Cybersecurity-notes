# Sharing Keys Safely: Asymmetric Encryption

## Opening Question

If Alice and Bob have never met and cannot safely send a secret key over the internet, how can they still exchange encrypted messages securely?

This problem is known as the **Key Distribution Problem**, and asymmetric encryption solves it.

---

# The Key Distribution Problem

In symmetric encryption:

* Alice and Bob use the **same key**
* The key encrypts and decrypts the data
* Encryption is very fast and efficient

However, there is a major issue:

> How do they safely share the secret key?

If the key is sent openly across the internet, an attacker can intercept it.

If they encrypt the key itself, they would need another secret key first, creating an endless cycle.

This problem led to the creation of **asymmetric encryption**.

---

# What is Asymmetric Encryption?

Asymmetric encryption uses **two mathematically related keys** instead of one.

## The Two Keys

### Public Key

* Can be shared with anyone
* Used for encryption

### Private Key

* Kept secret by the owner
* Used for decryption

---

# Core Principle

## Encrypt with Public Key → Decrypt with Private Key

If Alice encrypts a message using Bob's public key:

* Only Bob's private key can decrypt it
* Anyone can send Bob encrypted messages
* Nobody except Bob can read them

---

# Mathematical Security

The public and private keys are connected using complex mathematics.

Although the keys are related:

* It is computationally infeasible to derive the private key from the public key
* Modern computers would take hundreds or thousands of years to break strong keys

This computational difficulty provides security.

---

# Mailbox Analogy

Asymmetric encryption can be understood using a mailbox example.

## Public Key = Mail Slot

* Anyone can place letters into the mailbox
* Completely public and accessible

## Private Key = Mailbox Door Key

* Only the mailbox owner can open the box
* Used to retrieve and read messages

---

# Example Communication Flow

## Step 1 — Bob Creates Keys

Bob generates:

* A public key
* A private key

He keeps the private key secret.

---

## Step 2 — Bob Shares Public Key

Bob publishes his public key:

* Website
* Email
* Key server

Sharing it publicly is safe.

---

## Step 3 — Alice Encrypts Message

Alice:

* Downloads Bob's public key
* Encrypts her message using it
* Sends the encrypted data

---

## Step 4 — Bob Decrypts Message

Bob:

* Receives the encrypted message
* Uses his private key to decrypt it

Only Bob can read the message.

---

# Solving the Key Distribution Problem

The breakthrough of asymmetric encryption is:

> The secret key never needs to travel across the network.

Only the public key is shared.

Since the public key is not secret:

* Interception does not matter
* Attackers still cannot decrypt the messages

---

# Real-World Use: HTTPS

One of the most common uses of asymmetric encryption is HTTPS.

Whenever you see:

* A padlock icon
* `https://` in the browser

Asymmetric encryption is being used behind the scenes.

---

# What Happens During HTTPS?

## Step 1 — Browser Requests Public Key

Your browser contacts the website.

---

## Step 2 — Website Sends Certificate

The server sends:

* Its public key
* A digital certificate

---

## Step 3 — Shared Secret Creation

Using asymmetric encryption:

* Browser and server securely establish a shared symmetric key

---

## Step 4 — Switch to Symmetric Encryption

After setup:

* Symmetric encryption handles the rest of the communication
* This is faster and more efficient

---

# Hybrid Encryption

Modern systems combine both encryption types.

## Asymmetric Encryption

Used for:

* Secure key exchange
* Authentication
* Certificates

## Symmetric Encryption

Used for:

* Fast bulk data encryption
* Continuous communication

This combination is called a:

> Hybrid Encryption System

---

# Digital Certificates

A certificate is a digital document that proves ownership of a public key.

## A Certificate Contains

* Public key
* Website/domain name
* Certificate Authority signature
* Expiration dates

---

# Certificate Authorities (CAs)

A Certificate Authority is a trusted organization that verifies identities and signs certificates.

Examples include:

* DigiCert
* Let's Encrypt
* GlobalSign

Operating systems and browsers already trust many CAs by default.

---

# Browser Certificate Validation

When a website sends a certificate, the browser checks:

## 1. Trusted Signature

Was the certificate signed by a trusted CA?

## 2. Expiration

Is the certificate still valid?

## 3. Revocation Status

Has the certificate been revoked?

If all checks pass:

* Browser trusts the website
* Secure connection begins
* Padlock icon appears

If checks fail:

* Browser displays a warning
* Connection may be blocked

---

# Viewing Certificates in Your Browser

You can inspect certificates manually.

## Steps

1. Visit an HTTPS website
2. Click the padlock icon
3. Select:

   * "Connection is secure"
   * "Certificate"
   * "View certificate"

You can then view:

* Issued to
* Issued by
* Expiration dates
* Encryption details

# Symmetric vs Asymmetric Encryption

| Feature     | Symmetric Encryption | Asymmetric Encryption       |
| ----------- | -------------------- | --------------------------- |
| Keys Used   | One shared key       | Public + Private key        |
| Speed       | Very fast            | Slower                      |
| Key Sharing | Difficult            | Public key shared openly    |
| Main Usage  | Bulk data encryption | Key exchange & certificates |
| Example     | AES                  | RSA, ECC                    |
| Analogy     | One key opens lock   | Mailbox system              |

# Key Takeaways

* Symmetric encryption is fast but suffers from key distribution problems
* Asymmetric encryption solves secure key sharing
* Public keys can be shared safely
* Private keys must remain secret
* HTTPS combines both methods using hybrid encryption
* Certificates help verify identity and prevent impersonation attacks
